ロジカルシンキングとは何か

Posted on 2026년 3월 31일
#日本語 #就活 #ロジカルシンキング

# ロジカルシンキングとは

ロジカルシンキング（論理的思考）とは、物事を整理し、筋道を立てて考える思考法のことだ。 感情や直感に頼らず、根拠のある結論を導き出すことを目的とする。

ビジネスの現場では、以下のような場面で特に重要になる。

  • 問題の原因を正確に特定したいとき
  • 相手に納得感のある説明をしたいとき
  • 複雑な課題を整理して解決策を導き出したいとき

ロジカルシンキングは「賢さ」ではなく「技術」だ。訓練すれば誰でも身につけられる。


# 1. MECE（ミーシー）

Mutually Exclusive, Collectively Exhaustive 「モレなく、ダブりなく」という意味だ。

課題や情報を整理するとき、抜け漏れや重複があると、正確な判断ができない。 MECE はその状態を防ぐための基本的な考え方だ。

# 例：売上が下がった原因を考える

❌ MECEでない分け方
  └── 新規顧客が減った
  └── リピーターが減った
  └── 若い顧客が減った  ← 上の2つと重複する可能性がある

✅ MECEな分け方
  └── 新規顧客
  └── 既存顧客（リピーター）
      ← この2つで全顧客を網羅でき、重複もない

「他に抜けている視点はないか？」「この分類は重なっていないか？」 と常に問いかけることが MECE の実践だ。


# 2. ピラミッドストラクチャー

結論を先に述べ、その根拠を後から積み上げる構造のことだ。 日本語では「結論ファースト」とも呼ばれる。

        [結論]
       /      \
  [根拠①]  [根拠②]  [根拠③]
   /   \
[事実] [事実]

# なぜ重要か

人は結論が見えないまま話を聞くと、途中で「結局何が言いたいの？」となりやすい。 結論を先に伝えることで、聞き手は安心して根拠を受け取ることができる。

# 例

❌ 結論が後
「先月のユーザー数が20%減りました。
 また、解約率も上がっています。
 サポート対応の評価も下がっていました。
 なので、サービス品質に問題があると思います。」

✅ 結論が先（ピラミッド構造）
「サービス品質に問題があります。（結論）
 理由は3つあります。（根拠の数を先に示す）
 ①ユーザー数が20%減少、②解約率の上昇、③サポート評価の低下です。」

# 3. So What? / Why So?

思考を深めるための 2 つの問いかけだ。

問い 意味 使い方
So What? 「だから何？」 事実から意味・結論を引き出す
Why So? 「なぜそう言える？」 結論に根拠があるか確認する

# 例

事実：「ユーザーの平均セッション時間が2分から30秒に減った」

So What?（だから何？）
→「ユーザーがコンテンツに興味を持てていない可能性がある」

Why So?（なぜそう言える？）
→「セッション時間が短い = 離脱が早い = 魅力を感じていないと考えられるから」

So What? で結論を出し、Why So? で根拠を検証する。 この繰り返しが論理的な思考の訓練になる。


# 4. 5W1H

問題や課題を漏れなく把握するためのフレームワークだ。 特に課題の解像度を上げる最初のステップとして有効だ。

要素 意味 問いの例
Who 誰が 誰に影響が出ているか
What 何を 何が問題なのか
When いつ いつから発生しているか
Where どこで どの範囲で起きているか
Why なぜ なぜそれが問題なのか
How どのように どのように解決するか

# 例：「アプリのバグ報告が増えた」という課題

Who   → どのユーザー層から報告が多いか（新規？既存？）
What  → どの機能でバグが発生しているか
When  → いつから増えたか（特定のリリース後？）
Where → どの環境で発生しているか（iOS？Android？特定ブラウザ？）
Why   → なぜそのバグが発生しているか
How   → どう修正・再発防止するか

5W1H で問いを立てることで、思い込みや見落としを防ぐことができる。


# 5. 課題の解像度を上げる

ロジカルシンキングにおいて、課題を正確に定義することは解決策を考えることと同じくらい重要だ。 「課題の解像度が低い」まま進むと、的外れな解決策になってしまう。

# 課題の解像度が低い状態とは

❌ 解像度が低い課題
「ユーザーが離れている」

→ 誰が？いつから？どの機能で？なぜ？が不明
→ この状態で解決策を考えても的外れになる

# 課題を分解して解像度を上げる

MECE と 5W1H を使って課題を細かく分解していく。

「ユーザーが離れている」
    ↓ 分解（MECE）
├── 新規ユーザーの離脱
│   └── 登録フローで離脱？ → どのステップで？
└── 既存ユーザーの離脱
    └── 特定機能への不満？ → どの機能？いつから？

    ↓ 5W1Hで深掘り
「既存ユーザーが、先月のアップデート後から、
 通知機能に不満を持ち、週1回以上使っていた層が離脱している」

← これが解像度の高い課題定義

# 課題の解像度を上げるステップ

① 課題を言語化する（まず広く捉える）
      ↓
② MECEで分解する（モレなくダブりなく）
      ↓
③ 5W1Hで深掘りする
      ↓
④ So What? / Why So? で検証する
      ↓
⑤ 「誰が・何を・いつ・なぜ」が明確な課題文を作る

# 考察

ロジカルシンキングを学んで最初に感じたのは、 「自分はこれまでいかに曖昧に考えていたか」 ということだった。

例えば「なんかうまくいっていない」と感じたとき、 以前なら感覚的に対処しようとしていた。 でも 5W1H や MECE で分解してみると、 問題の輪郭がはっきり見えて、何をすべきかが自然と決まってくる。

エンジニアとしても、バグの原因調査や設計の意思決定など、 ロジカルシンキングが直接使える場面は多い。 「なぜこの設計にするのか」「どこに問題があるのか」を 筋道立てて説明できることは、チームで働く上で特に重要だと感じる。

ロジカルシンキングは特別な才能ではなく、 意識して使い続けることで身につく技術だと思う。 まずは日常の小さな問題から、5W1H や So What?を使って考える癖をつけていきたい。


まとめ：

  • MECE → モレなく、ダブりなく情報を整理する
  • ピラミッドストラクチャー → 結論ファーストで論理を組み立てる
  • So What? / Why So? → 事実と結論の間を検証する問いかけ
  • 5W1H → 課題を漏れなく把握するフレームワーク
  • 課題の解像度を上げる → 分解と深掘りで「本当の課題」を定義する
