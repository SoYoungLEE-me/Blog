# ✅ 1. 트렁크 기반 개발 (Trunk-Based Development)

# 배경 — 릴리즈 방식은 왜 회사마다 다를까?

팟캐스트에서 세 회사의 릴리즈 빈도가 전혀 다르다는 이야기가 나왔다.


회사 릴리즈 빈도 1회 규모
AbemaTV 2주에 1회 400~900 커밋 (빅뱅 릴리즈)
WinTicket 시책(기능) 완성 시마다 PR 10~20개
N-Organic 매일, 1인 1회 이상 매우 작음

이 차이가 생기는 근본적인 이유는 브랜치 전략이 다르기 때문이다.


# 일반적인 브랜치 전략

feature → dev → staging → production

대부분의 팀이 사용하는 방식이다. 피처 브랜치에서 개발한 내용을 dev, staging 브랜치에 순서대로 쌓아두다가 충분히 검증되면 한 번에 프로덕션에 올린다.

이건 나 역시 프로젝트를 진행할 때 많이 쓰는 방법이다. 하지만 프로젝트의 규모가 클 경우 적합한지는 한번 더 생각해봐야 한다.


  • 한 번에 많은 변경사항이 올라가기 때문에 릴리즈 규모가 커짐
  • QA를 한꺼번에 묶어서 처리할 수 있다는 장점이 있지만
  • 문제가 생겼을 때 어떤 변경이 원인인지 찾기 어려움

# 트렁크 기반 개발

feature → production (직접 머지 → 즉시 배포)

N-Organic이 사용하는 방식이다. dev/staging 브랜치 없이 피처 브랜치를 프로덕션 브랜치에 바로 머지하고, 머지되는 순간 자동으로 배포된다.


  • 변경 단위가 작기 때문에 문제 발생 시 원인을 빠르게 파악 가능
  • 엔지니어 전원이 하루 1회 이상 릴리즈하는 속도감이 가능해짐
  • 단, 언제든 프로덕션에 배포되기 때문에 자동화 테스트와 모니터링이 필수

구분 일반 브랜치 전략 트렁크 기반 개발
릴리즈 빈도 낮음 (주기적) 높음 (수시)
1회 릴리즈 크기 큼 작음
장점 QA를 묶어서 처리 가능 빠른 피드백, 작은 변경 단위
단점 문제 발생 시 원인 찾기 어려움 QA 자동화가 필수적으로 요구됨


# ✅ 2. 테스트 전략

# 배경 — 테스트가 왜 중요한가?

팟캐스트에서 세 회사 모두 수동 QA 비용이 너무 크다는 공통된 고민을 털어놨다. WinTicket은 라이브러리 버전 업 하나를 위해 수동 QA에 1주일이 걸리고, 그 기간이 이미 다른 QA로 막혀 있으면 한 달을 기다려야 하는 경우도 있다고 했다. 이 문제를 해결하기 위해 세 회사 모두 테스트 자동화를 목표로 하고 있다.


# 단위 테스트 (Unit Test)

함수나 컴포넌트 하나하나가 의도대로 동작하는지 확인하는 테스트다. Jest, React Testing Library 같은 도구를 사용한다.


// Jest + React Testing Library 예시
test("버튼을 클릭하면 카운트가 증가한다", () => {
  render(<Counter />);
  fireEvent.click(screen.getByText("+1"));
  expect(screen.getByText("1")).toBeInTheDocument();
});

세 회사 모두 단위 테스트를 어느 정도 작성하고 있지만, 모든 코드에 테스트가 있는 게 아니라 핵심 로직 위주로만 작성하고 있다고 했다.


오래된 코드일수록 테스트하기 어렵게 짜여 있는 경우가 많아서 테스트를 붙이려면 코드 구조 자체를 먼저 개선해야 하는 경우도 있다.


# E2E 테스트 (End-to-End Test)

실제 사용자처럼 브라우저를 직접 조작해서 전체 흐름을 검증하는 테스트다. 예를 들어 "로그인 → 상품 선택 → 결제 완료"처럼 유저 시나리오 전체를 자동으로 테스트한다.


  • 가장 현실적인 테스트지만 구축과 유지보수 비용이 크다
  • WinTicket은 현재 도입을 검토 중이며, 어떤 시나리오를 자동화할 수 있는지 조사 중
  • 결제처럼 외부 시스템과 연동된 기능은 E2E 자동화가 특히 어렵다

# VRT (Visual Regression Test, 시각적 회귀 테스트)

컴포넌트를 스크린샷으로 캡처해두고, 코드 변경 후 이전 캡처와 비교해서 의도하지 않은 UI 변경을 자동으로 감지하는 테스트다.

리팩토링이나 라이브러리 버전 업 후 "겉으로 보기엔 멀쩡한데 UI가 살짝 깨진" 경우를 사람이 일일이 확인하지 않아도 자동으로 잡아낸다.


# 기존 스냅샷 테스트의 문제점

VRT 이전에는 스냅샷 테스트를 많이 사용했다. HTML 구조를 텍스트로 저장해두고 변경이 생기면 알려주는 방식인데, 텍스트 차이만 보여주기 때문에 "이게 실제로 화면이 어떻게 바뀐 건지" 알기 어렵다는 문제가 있었다.


expect(component).toMatchSnapshot();
// 차이가 생겼을 때 "이게 의도한 변경인가?" 판단이 어려움
// → 확인도 안 하고 그냥 업데이트해버리는 경우가 많아 의미가 퇴색됨

# VRT의 장점

  • 이전/이후 스크린샷을 시각적으로 비교할 수 있어 의도한 변경인지 판단이 쉬움
  • Storybook과 함께 사용하면 컴포넌트 단위로 관리 가능
  • 주요 툴: Chromatic

# Flaky 테스트 문제와 해결

VRT의 가장 큰 단점은 이미지가 포함된 컴포넌트에서 픽셀 단위 차이로 오탐(false positive, 실제로는 문제없는데 차이가 있다고 감지하는 것)이 자주 발생한다.


❌ 문제: 실제 이미지를 사용하면 압축/해상도 차이로 매번 다른 결과가 나옴
✅ 해결: 테스트용 단색 이미지로 교체 → 픽셀 차이 자체를 없애버림

AbemaTV는 기존 스냅샷 테스트에서 VRT로 전환하는 데 약 6개월이 걸렸다. 처음부터 도입하는 것이 훨씬 수월하다는 교훈을 얻었다고 한다.



# ✅ 3. Critical CSS 추출

# 배경 — 왜 필요한가?

웹페이지가 처음 열릴 때 브라우저는 CSS 파일을 다운로드할 때까지 화면을 그리지 못한다. CSS 파일이 크거나 네트워크가 느리면 흰 화면이 오래 보이는 문제가 생긴다.


# 개념

첫 화면(Above the Fold, 스크롤 없이 보이는 영역)에 필요한 CSS만 추출해서 HTML 파일 안에 직접 집어넣는 방식이다. 이렇게 하면 별도 CSS 파일 로드를 기다리지 않아도 첫 화면이 바로 그려진다.


<!-- 일반적인 방식: CSS 파일이 로드될 때까지 화면을 그리지 못함 -->
<link rel="stylesheet" href="/styles.css" />

<!-- Critical CSS 적용: 첫 화면에 필요한 CSS가 HTML 안에 바로 포함 -->
<style>
  header {
    ...;
  }
  .hero {
    ...;
  }
</style>
<link rel="stylesheet" href="/styles.css" />
<!-- 나머지 CSS는 비동기로 로드 -->

# CI에서의 처리 흐름

Critical CSS를 추출하려면 실제 브라우저로 페이지를 열어봐야 하기 때문에 CI 파이프라인 안에서 Chrome을 직접 실행한다.


CI에서 Chrome 실행
    ↓
실제 페이지 캡처
    ↓
첫 화면에 필요한 CSS만 추출
    ↓
HTML에 인라인으로 삽입
    ↓
배포

# 트레이드오프

성능은 좋아지지만 그만큼 빌드 과정이 복잡해진다. AbemaTV의 경우 이 과정이 전체 배포 시간을 늘리는 주요 원인 중 하나라고 했다.


장점 단점
초기 렌더링 속도 향상 CI에서 Chrome을 실제로 띄워야 해서 빌드 시간 증가
CSS 로드 대기 시간 제거 구현 및 유지보수 복잡도 증가


# ✅ 4. Sentry 운용 노하우

# 배경 — Sentry가 뭔가?

Sentry는 프로덕션에서 에러가 발생했을 때 자동으로 감지해서 알려주는 모니터링 도구다. 세 회사 모두 Sentry를 사용하고 있었고, 공통적으로 노이즈 관리가 가장 큰 고민이었다.

노이즈란 별로 중요하지 않은 에러 알림이 너무 많이 쌓여서 정작 진짜 문제가 생겼을 때 알아채지 못하는 상황을 말한다.


# 릴리즈 태그 활용

Sentry에 릴리즈 버전을 태그해두면, 어떤 배포 이후에 에러가 급증했는지 바로 파악할 수 있다.


Sentry.init({
  dsn: "...",
  release: "my-app@1.0.0", // 배포할 때마다 버전을 업데이트
});
// → "1.0.0 배포 이후에 이 에러가 갑자기 늘었다" 는 걸 바로 알 수 있음

# 노이즈를 줄이는 핵심 원칙

팟캐스트에서 가장 인상적이었던 내용이다. 노이즈를 줄이는 근본적인 방법은 애플리케이션 코드에서 에러 핸들링을 제대로 하는 것이다.

예를 들어 "유저를 찾을 수 없음(404)"은 충분히 예상 가능한 상황이다. 이런 에러까지 Sentry에 보내면 노이즈만 쌓인다.


// ❌ 예상 가능한 에러까지 전부 Sentry에 전송
try {
  const user = await fetchUser(id);
} catch (err) {
  Sentry.captureException(err); // 404도 전송되어 노이즈가 됨
}

// ✅ 진짜 예상치 못한 에러만 전송
try {
  const user = await fetchUser(id);
  if (!user) return null; // 404는 여기서 처리
} catch (err) {
  if (err.status !== 404) {
    Sentry.captureException(err); // 진짜 문제만 Sentry로
  }
}

# 운용 팁

  • 노이즈성 에러는 발견하는 즉시 ignore 처리
  • 방치하면 진짜 에러를 놓치고 Sentry 과금도 늘어남
  • 팀에서 정기적으로 에러를 정리하는 루틴을 만드는 것이 효과적


# ✅ 5. 빌드 최적화 기법

# 배경 — 빌드가 왜 느린가?

팟캐스트에서 N-Organic의 프론트엔드 빌드가 20분이나 걸린다는 게 화제가 됐다. WinTicket은 5분, AbemaTV는 빌드 자체는 3분이지만 전체 CI는 20분이라고 했다. 빌드가 느리면 PR 리뷰 사이클도 느려지고, 개발 속도 전체가 떨어진다.


# thread-loader — Webpack 병렬 빌드

Webpack은 기본적으로 파일을 하나씩 순서대로 변환한다. thread-loader를 사용하면 여러 파일을 동시에 병렬로 처리할 수 있다. CI 머신의 CPU 코어 수에 맞게 스레드를 설정하는 게 포인트다.


// webpack.config.js
module.exports = {
  module: {
    rules: [
      {
        test: /\.jsx?$/,
        use: [
          "thread-loader", // 이후 로더를 별도 스레드에서 병렬 실행
          "babel-loader",
        ],
      },
    ],
  },
};

# Docker 레이어 캐시 최적화

Docker는 빌드할 때 변경이 없는 레이어는 캐시를 재사용한다. 파일 순서를 잘못 배치하면 캐시가 매번 깨져서 처음부터 다시 빌드하게 된다.



❌ 소스코드를 먼저 복사하면 코드가 바뀔 때마다 npm install도 다시 실행됨

COPY . .
RUN npm install

✅ 변경이 적은 파일을 먼저 → npm install은 package.json이 바뀔 때만 재실행

COPY package.json package-lock.json ./
RUN npm install # package.json이 안 바뀌면 이 레이어는 캐시 재사용
COPY . . # 소스코드는 여기서만 캐시 무효화

WinTicket은 Dockerfile을 이 구조로 개선한 후 배포 시 빌드 시간이 크게 단축됐다고 했다.


핵심 요약:

  • 트렁크 기반 개발 → 피처 브랜치를 프로덕션에 직접 머지, 작은 단위로 자주 배포
  • 단위 테스트 / E2E 테스트 → 수동 QA 비용을 줄이기 위한 자동화 목표, 핵심 로직부터
  • VRT → 스크린샷 비교로 의도치 않은 UI 변경 자동 감지, 처음부터 도입할수록 유리
  • Critical CSS → 첫 화면 렌더링 속도 향상, 빌드 시간 증가라는 트레이드오프 존재
  • Sentry → 예상 가능한 에러는 코드에서 처리하고, 진짜 에러만 전송해 노이즈 줄이기
  • 빌드 최적화 → thread-loader로 병렬 처리, Docker 레이어 순서로 캐시 재사용
