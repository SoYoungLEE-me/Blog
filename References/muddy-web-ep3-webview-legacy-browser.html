WebView와 레거시 브라우저 대응 - 개발 효율화 노하우

Posted on 2026년 3월 31일
#Frontend #WebView #Browser #Testing #Linter

# 출처

🎙️ マディウェブポッドキャスト EP.3 — レガシーブラウザは IE だけじゃない！？ABEMA 試合データ機能で培った WebView 開発ノウハウ
CyberAgent AbemaTV 소속 프론트엔드 엔지니어가 월드컵 기능 개발 당시 WebView 도입과 레거시 브라우저 대응 과정에서 겪은 경험을 이야기한 팟캐스트를 듣고 기술적으로 배울 만한 내용을 정리한다. 🔗 Spotify에서 듣기 (opens new window)



# ✅ 1. 레거시 브라우저 조사 — 공식 문서에 없으면 직접 파헤친다

# 배경 — 왜 조사가 필요했나?

담당 엔지니어는 AbemaTV에서 월드컵 시청 중 실시간 경기 데이터를 함께 보여주는 기능을 개발했다고 한다.


AbemaTV 월드컵 기능은 웹과 앱 양쪽에서 제공해야 했는데, 각각 따로 만들면 공수가 너무 크기 때문에 WebView(앱) + iFrame(브라우저) 으로 코드를 공통화했다고 한다.


WebView란? 앱 안에 웹 브라우저를 내장해서 웹 페이지를 보여주는 기능이다. 네이티브 앱(iOS/Android) 안에서 웹으로 만든 화면을 그대로 띄울 수 있어서, 웹과 앱 양쪽에서 같은 코드를 재사용할 수 있다는 장점이 있다.


WebView는 기기마다 탑재된 브라우저 버전이 달라서 어떤 버전까지 지원해야 하는지 먼저 파악하는 게 출발점이었다고 했다. Android, iOS는 공식 문서에 명시되어 있어서 쉬웠지만 Fire Tablet은 달랐다.


Fire Tablet이란? Amazon이 만든 태블릿 기기로, 일반 Android와 달리 Amazon이 커스터마이징한 OS를 사용한다. AbemaTV의 지원 대상 기기에 포함되어 있었는데, 내부적으로 어떤 브라우저 엔진을 쓰는지 공식 문서에 자세히 나와있지 않아서 조사가 까다로웠다고 한다.


  • Chromium 기반이라는 것만 공식 문서에 나와있고
  • 어떤 버전인지는 아무 데도 적혀있지 않았다

# 해결 방법

팟캐스트에서 담당 엔지니어는 Fire Tablet OS 업데이트 파일을 직접 분석해서 내부에 설치된 WebView 버전을 추출하는 스크립트를 직접 작성해서 해결했다고 한다.


Fire Tablet OS 업데이트 파일 다운로드
    ↓
내부 앱 목록 추출 스크립트 실행
    ↓
WebView 버전 확인
    ↓
해당 Chromium 버전의 기능 지원 범위 파악

조사해보니 생각보다 훨씬 오래된 버전이 탑재되어 있었다고 했다. 미리 확인하지 않았다면 배포 후에 큰 문제가 됐을 것이라는 이야기가 인상적이었다.

공식 문서에 없는 정보라도 포기하지 않고 직접 파헤치는 태도가 인상적이었다. 나도 막히는 부분이 생겼을 때 "문서에 없으니까 모른다"가 아니라 직접 찾아내는 방법을 고민하는 습관을 들이고 싶다.



# ✅ 2. 커스텀 Linter로 개발 실수 방지

# 배경 — 왜 Linter가 필요했나?

Linter란? 코드를 작성할 때 문법 오류나 규칙 위반을 자동으로 감지해서 알려주는 도구다. ESLint(JavaScript), Stylelint(CSS) 등이 대표적이다. 저장할 때마다 자동으로 검사해주기 때문에 실수를 개발 중에 바로 잡을 수 있다.


오래된 브라우저에서 동작하지 않는 CSS 속성을 무의식 중에 써버리는 실수가 자주 발생했다고 한다. 예를 들어 CSS Grid는 오래된 브라우저에서 지원되지 않는데, 최신 브라우저로 개발하다 보면 자연스럽게 쓰게 된다.


팟캐스트에서는 이걸 배포 후에 발견하면 고치는 비용이 크기 때문에 개발 중에 자동으로 잡을 수 있는 방법을 고민했다고 했다.


# 해결 방법 — Stylelint 커스텀 플러그인 자작

MDN은 "이 CSS 속성은 어떤 브라우저 버전부터 지원된다"는 호환성 정보를 JSON 패키지 형태로 공개하고 있다.
담당 엔지니어는 이 데이터를 재료로 삼아, 지원 불가 CSS를 자동으로 감지하는 Stylelint 플러그인을 직접 만들었다고 했다.


MDN 브라우저 호환성 데이터 (JSON 패키지로 제공됨)
    ↓
커스텀 Stylelint 플러그인
    ↓
사용 불가 CSS 작성 시 → 즉시 에러
    ↓
개발 중 로컬에서 즉시 피드백

이 플러그인 덕분에 레거시 브라우저를 직접 켜서 확인하지 않아도 됐고, 리뷰에서 같은 지적이 사라져 리뷰 코스트도 크게 줄었다고 했다.


필요한 도구가 없으면 직접 만든다는 발상이 인상적이었다. Linter는 단순히 코드 스타일을 맞추는 것뿐 아니라 이런 식으로 팀의 실수를 구조적으로 막는 역할도 할 수 있다는 걸 배웠다.



# ✅ 3. 반복되는 리뷰 지적은 Linter로 자동화

# 배경

CSS 변수를 사용해야 하는 규칙이 있었는데, 개발 중에 직접 px 값을 하드코딩하는 실수가 반복됐다고 한다.


/* ❌ 직접 값을 넣어버리는 실수 */
margin: 8px;

/_ ✅ CSS 변수를 사용해야 하는 올바른 방식 _/
margin: var(--spacing-2);

리뷰에서 같은 지적이 몇 번이나 반복되자 이것도 Linter 규칙으로 자동화했다고 했다.

"같은 리뷰 지적이 반복된다면, 그건 자동화할 수 있다는 신호다." 이 말이 굉장히 와닿았다. 사람이 매번 지적하는 것보다 도구가 자동으로 잡아주는 것이 훨씬 효율적이고, 무엇보다 리뷰어도 개발자도 서로 스트레스를 받지 않는다.



# ✅ 4. Docker로 레거시 브라우저 확인 환경 구축

# 배경

Linter로 어느 정도 막을 수 있지만, 실제로 레거시 브라우저에서 화면이 어떻게 보이는지 직접 확인할 수 있는 환경도 필요했다고 한다.



Docker 안에 오래된 Chromium 설치
↓
VNC로 Docker 안의 가상 화면에 접속
↓
오래된 Chromium에서 Storybook 열어서 확인

가장 오래된 지원 대상은 Android 5의 WebView = Chromium 37 (2014년) 이었다고 한다. Storybook이 최신 브라우저 기준으로 만들어져 있어서 동작하지 않는 부분은 몽키패치로 직접 수정해서 사용했다고 했다.


몽키패치(Monkey Patch)란? 원본 코드를 수정하지 않고 런타임에 임시로 동작을 덮어씌우는 방식이다. 라이브러리가 레거시 브라우저에서 동작하지 않을 때 임시 대응으로 사용한다.


// 레거시 브라우저에 없는 메서드를 임시로 추가하는 예시
if (!Array.prototype.at) {
  Array.prototype.at = function (index) {
    return index >= 0 ? this[index] : this[this.length + index];
  };
}


# ✅ 5. VRT로 스타일 깨짐 자동 감지

레거시 브라우저에서의 스타일 깨짐을 자동으로 감지하기 위해 Storybook + VRT(Visual Regression Test)를 함께 도입했다고 한다.


VRT(Visual Regression Test, 시각적 회귀 테스트)란? 컴포넌트를 스크린샷으로 캡처해두고, 코드 변경 후 이전 캡처와 비교해서 의도하지 않은 UI 변경을 자동으로 감지하는 테스트 방법이다. 리팩토링이나 라이브러리 버전 업 후 "겉으로 보기엔 멀쩡한데 UI가 살짝 깨진" 경우를 사람이 일일이 확인하지 않아도 자동으로 잡아낼 수 있다.


오래된 Chromium에서 Storybook 렌더링
    ↓
전체 컴포넌트 스크린샷 캡처
    ↓
이전 스크린샷과 차이 비교
    ↓
❌ 차이 있음 → PR에서 즉시 확인
✅ 차이 없음 → 정상

모든 브라우저에서 찍으면 비용이 너무 크기 때문에 지원 대상 중 가장 오래된 버전(Chromium 37) 하나만 기준으로 삼았다고 했다. PR 리뷰 시 스크린샷만 봐도 레거시 브라우저에서의 상태를 바로 파악할 수 있어서 개발 효율이 크게 올라갔다고 한다.


"실제로 해보니 무너진 UI가 꽤 많았다. 준비해두길 정말 잘했다."


EP.1에서 AbemaTV가 VRT를 전체 프로젝트에 도입하는 데 6개월이 걸렸다는 이야기를 들었는데, 이 에피소드에서도 VRT가 레거시 브라우저 대응에 실질적으로 효과가 있었다는 걸 확인했다. 처음부터 도입해두는 것이 얼마나 중요한지 다시 한번 느꼈다.



# ✅ 6. 테스트 환경 이중화 — Node.js vs 실제 브라우저

단위 테스트를 Node.js 환경과 실제 브라우저 환경 두 곳에서 모두 실행할 수 있도록 구성했다고 한다.


구분 Node.js (Jest 등) 브라우저 (Karma)
속도 빠름 느림
환경 가상 DOM (JSDOM) 실제 브라우저
사용 시점 로컬 개발 중, PR CI 릴리즈 전 최종 검증

Node.js 환경은 가상 DOM을 사용하기 때문에 외부 라이브러리가 레거시 브라우저에서 실제로 동작하는지 알 수 없다. 브라우저에서 테스트를 돌리면 라이브러리 내부에서 필요한 폴리필이 없는 경우도 감지할 수 있다고 한다.


테스트 코드는 동일
    ↓
실행 환경만 전환
  ├── Node.js  → 빠른 피드백 (로컬 개발 중, PR CI)
  └── 브라우저 → 정확한 검증 (릴리즈 전)


# ✅ 7. 모던/레거시 브라우저에 다른 번들 제공

# 배경

폴리필을 모든 사용자에게 다 제공하면 번들 크기가 불필요하게 커진다. 최신 브라우저를 쓰는 사람에게도 필요 없는 코드를 내려보내는 건 낭비라고 했다.


폴리필(Polyfill)이란? 오래된 브라우저에서 최신 기능이 동작하도록 해당 기능을 JS로 직접 구현해서 채워 넣는 코드다.


ESModule 지원 여부를 기준으로 번들을 분리 제공하는 방식을 도입했다고 한다.


<!-- 오래된 브라우저 → 폴리필 포함된 무거운 번들 -->
<script nomodule src="legacy.bundle.js"></script>

<!-- 최신 브라우저 → 폴리필 없는 가벼운 번들 -->
<script type="module" src="modern.bundle.js"></script>

  • type="module" → ESModule 지원 브라우저만 읽음
  • nomodule → ESModule 미지원 브라우저만 읽음
  • 브라우저가 자신에게 맞는 번들을 알아서 선택

# Web API 폴리필 분리 문제

@babel/preset-env는 ECMAScript 폴리필(core-js)만 처리할 수 있고 fetch 같은 Web API 폴리필은 브라우저별로 분리 제어가 불가능하다는 한계가 있었다고 한다.

이 문제도 MDN 호환성 데이터를 활용한 Babel 플러그인을 직접 작성해서 해결했다고 했다.



MDN Web API 호환성 데이터
↓
커스텀 Babel 플러그인
↓
지원 브라우저에 따라 Web API 폴리필 자동 추가/제거


이 에피소드 전체를 들으면서 느낀 건 "없으면 만든다" 는 태도였다. 공식 문서에 없는 정보는 스크립트로 직접 파헤치고, 필요한 Linter 플러그인이 없으면 직접 만들고, Babel 플러그인도 필요하면 직접 작성했다.

단순히 있는 도구를 가져다 쓰는 것이 아니라 문제를 정확히 정의하고 그 문제를 해결하는 도구를 만드는 것, 그게 엔지니어로서 진짜 실력이라는 걸 느꼈다.

반복되는 리뷰 지적을 Linter로 자동화하는 사고방식도 배울 점이 많았다. "또 같은 걸 지적해야 하네"가 아니라 "이건 자동화할 수 있겠다"로 연결하는 것, 이런 사고방식을 평소 개발에서도 의식하고 싶다.


핵심 요약:

  • 레거시 브라우저 조사 → 공식 문서에 없으면 직접 스크립트로 분석
  • 커스텀 Linter → MDN 데이터 기반으로 사용 불가 CSS를 개발 중 즉시 감지
  • 반복 지적 → Linter 자동화 → 같은 지적이 반복되면 자동화할 수 있다는 신호
  • VRT → 가장 오래된 브라우저 기준 스크린샷 비교로 스타일 깨짐 자동 감지
  • 테스트 이중화 → Node.js(빠른 피드백) + 브라우저(정확한 검증) 용도 분리
  • 번들 분리 → type="module" / nomodule로 모던/레거시 브라우저에 맞는 번들 제공
